# Wipster — Product Requirements Document (PRD)

## 1. Обзор продукта

**Wipster** — персональная система управления задачами и продуктивностью с жёстким WIP-лимитом, автоматическим захватом контекста, обнаружением отвлечений и ИИ-агентом для массовых операций и приоритезации.

**Платформы:** macOS desktop (кроссплатформенный фреймворк: Tauri или Electron) + Telegram-бот.

**Целевой пользователь:** человек, который легко загорается новыми задачами, глубоко погружается в работу, но теряет фокус при переключении; склонен к параллельному ведению 6–10 задач; не любит бюрократию и лишний ввод.

---

## 2. Проблемы, которые решает продукт

| Проблема | Как решает Wipster |
|---|---|
| Слишком много параллельных задач (6–10) | Жёсткий WIP-лимит = 3 |
| Потеря контекста при переключении | Автоматический Return-anchor (app/URL/файл) |
| Трудно вернуться к задаче после паузы | Resume восстанавливает контекст в 1 клик |
| Потеря интереса и переключение на новое | Правило: новое — только через слот (Swap или Queue) |
| Мелочь/админка/хвосты копятся | 2 фиксированных "хвостовых" окна в неделю |
| Нет системы планирования | Triage-режим + Weekly Review + ИИ-приоритезация |
| Лень вводить данные | Минимум ввода: только название; всё остальное — клики и автозахват |

---

## 3. Ключевые принципы

1. **Минимум ввода** — пользователь вводит только название задачи. Всё остальное: клики, автодогадки, автозахват контекста.
2. **WIP = 3** — жёсткий лимит задач в статусе Doing. Попытка начать 4-ю → диалог Swap.
3. **Return-anchor** — при паузе/переключении автоматически сохраняется контекст (приложение, URL, файл, git-ветка).
4. **Агент не болтает** — только блокирующие моменты, preview → apply → undo, быстрые кнопки.
5. **Уточнения — кнопками** — если агенту нужно уточнение, он предлагает 2–6 вариантов, а не открытый вопрос.
6. **Приватность** — контекст хранится локально, скрины не хранятся по умолчанию, чувствительные приложения исключены из захвата.
7. **≤ 2 кликов** — любое уточнение/действие решается максимум в 2 клика в типичном случае. Ручной ввод — всегда последний вариант, не основной.

## 4. Функциональные требования

### 4.1. Управление задачами (Kanban + Projects)

- **Проекты** — список проектов слева; внутри каждого проекта: Queue / Doing / Done.
- **"All Doing"** — глобальное представление всех 3 активных задач из всех проектов.
- **Someday/Maybe** — отдельный список для идей, чтобы не засоряли backlog.
- **Quick Add** — одна строка + Enter. Автопредложение проекта по контексту.
- **Статусы:** `inbox → queue → doing → done`.
- **Переключение статуса** — перетаскиванием или 1 кликом.

### 4.2. WIP-лимит (= 3)

- Попытка `Start` на 4-ю задачу → диалог:
  - Swap: выбрать одну из Doing → обратно в Queue.
  - Keep in Queue: новая задача остаётся в очереди.
- Это 1–2 клика, без текста.

### 4.3. Карточка задачи

Поля задачи (все, кроме title, заполняются кликами или автоматически):

| Поле | Способ заполнения |
|---|---|
| Title | Ручной ввод (единственное обязательное) |
| Project | 1 клик, автопредложение по контексту |
| Status | inbox / queue / doing / done — клик/drag |
| Priority | P0–P3 или Low/Med/High — 1 клик |
| Due (срок) | Быстрые пресеты: Today / This week / Date-picker |
| Estimate | S / M / L — 1 клик |
| Tags | Автопредложение + 1 клик |
| Dependencies | "Blocked by…" / "Blocks…" — клик |
| Context | Автозахват (app/tab/file) + кнопка "Attach current" |
| DoD (Definition of Done) | 1 строка; обязательна при переводе в Doing |
| Checklist (acceptance criteria) | Опционально, 3–7 галочек; ИИ предлагает по кнопке |
| Next step | 1 строка; обязательна при Doing/Pause |
| Return ref | Автозахват контекста при Pause |

**Два уровня "готово"** (против перфекционизма):
- **Done (достаточно)** — минимальные обязательные критерии.
- **Polish (опционально)** — улучшения, не блокирующие Done.

### 4.4. Return-anchor (контекст при переключении)

При нажатии **Pause** / **Switch** у задачи, приложение автоматически фиксирует:
- Активное приложение + заголовок окна.
- URL (если браузер) — через расширение.
- Путь к файлу/проекту в IDE — через extension.
- Git-ветка/репо.

Кнопка **Resume** восстанавливает контекст (открывает ссылку/файл/приложение).

### 4.5. Автозахват контекста (macOS)

| Источник | Метод |
|---|---|
| Активное приложение + title окна | macOS Accessibility API (AXUIElement) |
| Браузерный таб (URL + title) | Расширение для Comet (Chromium-based) |
| Файл/проект/git в IDE | Расширение для Cursor (VS Code extension API) |
| Git-ветка/репо | Через IDE extension или CLI |

**Режим "Sensitive apps":** менеджер паролей, банки, мессенджеры — исключены из захвата.

### 4.6. Обнаружение отвлечений

Принцип: event-driven, не постоянный скрин-стриминг.

**Сигналы:**
- Активное приложение + title окна.
- URL/домен активной вкладки.
- Текущий workspace/файл в Cursor.

**Правило:**
- У активной задачи есть "допустимые контексты" (приложения/домены/репо).
- Если N секунд (настраиваемо: 30 сек / 1–2 мин / 5 мин) вне допустимого контекста → уведомление:
  - **Resume** — вернуть фокус на текущую задачу.
  - **Switch task** — переключить задачу правильно.
  - **Snooze 15m**.

### 4.7. Next-step suggestions (ИИ)

**Когда генерировать:**
- При Pause задачи.
- При Resume после долгого перерыва.
- Когда задача висит N дней без прогресса.

**На основе:**
- Название задачи, DoD, чеклист.
- Последние контекст-снимки (файл/URL/title).
- Последние коммиты/дифф (если есть).

**Формат:**
- 3 варианта next step (коротко).
- Кнопка "Принять" (сохраняет как Next).
- Кнопка "Не сейчас".

### 4.8. Ежедневный ритуал (Daily Focus, ~3 минуты)

Утром (или перед работой):
1. Показать текущие Doing (3 задачи).
2. Выбрать **одну** на сегодня как главную.
3. Предложить поставить 1 слот в календарь (60–90 минут).

Без списков на 30 пунктов. Один экран, 2–3 клика.

### 4.9. "Уродливая первая версия" (для неприятных/туманных задач)

Когда задача помечена как неприятная или с неопределённостью:
- Агент предлагает режим **"Черновик за 15–30 минут"** — сделать криво, но по структуре.
- После черновика задача автоматически получает next step "Улучшить черновик".
- Цель: снять стартовое трение.

### 4.10. Triage-режим (систематизация "груды")

Отдельный режим на 10–15 минут:
- Показывает задачи из inbox **по одной**.
- На каждую: 2–3 клика — Project + Priority + Deadline + Estimate + Tag + Dependency.
- Кнопка **Skip** (без наказания).
- Цель: быстро превратить inbox в структурированный backlog.

### 4.11. Приоритезация

Переключаемые режимы:
- **Deadline-first** — сроки/обязательства.
- **Impact-first** — ценность/эффект.
- **Unblock-first** — разблокировать цепочки зависимостей.
- **Energy-first** — прогресс через интерес.

Агент показывает **Top-5 на неделю** и **Top-3 Doing**, с коротким обоснованием на каждую задачу ("дедлайн", "разблокирует X", "высокий эффект при S").

### 4.12. Weekly Review

15–25 минут, с кнопками и агентом:
1. Inbox → Triage (проект/приоритет/срок/skip).
2. Queue → выбрать Top-5 на неделю.
3. Doing → убедиться, что максимум 3 и у каждой есть next/return.
4. "Хвосты" → запланировать 2 окна на закрытие мелочей.

### 4.13. "Хвостовые" окна (Tail Sessions)

2 фиксированных окна в неделю (например, вт/пт по 45–60 мин):
- Закрытие мелочей, админки, коммуникаций.
- Агент собирает подходящие задачи автоматически.
- Без этих окон "всё может зависнуть" будет повторяться.
- Кнопка **"Tail session"** запускает режим массового закрытия мелочей.

### 4.14. Интеграция с календарём

- **Google Calendar API / CalDAV** — двусторонняя интеграция.
- Создание слотов "Focus: <task>" из приложения.
- Обнаружение конфликтов: если созвон влезает в deep-work слот → агент предлагает перенос.
- Режим "не отвлекать" при активном созвоне (из календаря).

### 4.15. Голосовой ввод контекста (опционально, позже)

- Кнопка "Сказать контекст голосом" → speech-to-text → агент извлекает поля задачи (проект, срок, зависимости).
- Альтернатива ручному заполнению для сложных вводных.

### 4.16. Definition of Done + Checklist (ИИ-генерация)

- По кнопке **"Добавить критерии"** (не автоматически всегда).
- ИИ предлагает:
  - 1 строку DoD.
  - 3–7 галочек.
- Пользователь выбирает: "Принять все" / "Оставить ключевые" / "Убрать лишнее".
- Для повторяющихся типов задач — сохранять как **шаблон**.

## 5. Режим агента (Agent Mode)

Доступен и в **desktop**, и в **Telegram**. Предназначен для сложных изменений и массовых операций.

### 5.1. Принцип работы

1. Пользователь пишет запрос на естественном языке.
2. Агент уточняет сущности (если нужно — кнопками, 2–6 вариантов).
3. Агент формирует **план действий** (последовательность tool-вызовов).
4. Показывает **preview изменений** (diff).
5. Пользователь нажимает **Apply** → изменения применяются.
6. Доступен **Undo**.

### 5.2. Набор tools (единый API для desktop и Telegram)

**Поиск/чтение:**
- `list_projects()`
- `search_tasks(query, filters, limit)`
- `get_task(id)`
- `get_context_snapshot(latest | by_task)`

**CRUD:**
- `create_task(title, project_id?, ...)`
- `update_task(id, fields)`
- `bulk_update(ids, fields)`
- `move_tasks(ids, project_id?, column)`
- `set_dependencies(edges[])`
- `close_tasks(ids)` / `reopen_tasks(ids)`

**Планирование:**
- `suggest_priorities(scope, mode)`
- `schedule_focus_blocks(task_id, rules)`
- `create_triage_session(scope)` / `triage_next()` / `triage_apply()`

**Безопасность/надёжность:**
- `dry_run(plan)` → diff
- `apply(plan_id)`
- `undo(change_id | last)`
- `audit_log(limit)`

### 5.3. Протокол уточнений

- Не спрашивать, если можно вывести из данных (последний проект, последняя задача, контекст окна).
- Один вопрос → один экран → 2–6 вариантов.
- Всегда есть: быстрые кнопки + "Other..." + "Skip".
- Дефолт отмечать первой кнопкой.
- При уверенности > 0.85 → предложение по умолчанию + кнопки "Ок / Изменить".

### 5.4. Формат структурированного ответа агента

Агент возвращает не просто текст, а структуру:

```
{
  "summary": "Короткое резюме (1–2 строки)",
  "preview": [...],           // diff: что у каких задач меняется
  "question": "...",          // если нужно уточнение
  "options": [...],           // 2–6 вариантов ответа
  "option_actions": [...],    // tool-вызов для каждого варианта
  "default_option": 0,        // индекс дефолта
  "buttons": ["Apply", "Refine...", "Pick...", "Undo", "Cancel"]
}
```

Кликом по опции приложение вызывает соответствующий tool.

### 5.5. Права выполнения

- По умолчанию: может менять status, project, priority, due, tags, dependencies.
- "Опасные" действия (удаление, массовое закрытие, перенос между проектами) → отдельное подтверждение.

## 6. Telegram-бот (мобильный клиент)

### 6.1. Роль

"Мобильный пульт": быстрый inbox, уведомления, быстрые действия, полноценный агентный режим.

### 6.2. Основные сценарии

**Добавить задачу:**
- Пользователь: `Найти подход к X`
- Бот: "Куда?" + кнопки проектов (последние 5) + "В inbox"

**WIP-контроль (Start при 3 Doing):**
- Бот: "У тебя уже 3 Doing. Что сделать?"
- Кнопки: Swap (задача1) / Swap (задача2) / Swap (задача3) / Оставить в Queue

**Triage:**
- Бот показывает задачи по одной.
- Кнопки: Project / Priority / Deadline / Estimate / Skip / Done

**Агентные массовые изменения:**
- Пользователь: "Перенеси всё про интеграцию в проект Integrations и поставь дедлайн на следующую неделю"
- Бот: план + diff → кнопки: Apply / Изменить фильтр / Отмена / Undo

**Быстрые команды:** `/doing`, `/queue`, `/triage`, `/top`, `/undo`

### 6.3. Ограничение приватности

- Не отправлять в Telegram чувствительные данные.
- Контекст экрана/скрины не тащить в Telegram; максимум — короткие метаданные.

## 7. Desktop-приложение (macOS-first)

### 7.1. Форм-фактор

- **Menu bar app** с глобальной горячей клавишей.
- Кроссплатформенный фреймворк (Tauri или Electron).
- Управление кликом/drag&drop.

### 7.2. Структура UI

- **Слева:** список проектов.
- **Центр:** Kanban (Queue / Doing / Done) для выбранного проекта.
- **"All Doing":** глобальное представление 3 активных задач.
- **Справа (по необходимости):** Agent panel — ввод текста → план → preview → Apply → Undo.

### 7.3. Горячие клавиши

- Глобальная: открыть/скрыть Wipster.
- Quick Add.
- Start / Pause / Done.

### 7.4. Deep-work слоты

- Минимум 3 блока в неделю по 90–120 минут.
- В эти блоки — только 1 задача из "в работе".
- Подробности интеграции с календарём — см. раздел 4.14.

---

## 8. Модель данных

```
Project(id, name, order)

Task(id, title, project_id, status: inbox|queue|doing|done,
     priority, due, estimate, tags[],
     dod, checklist[], next_step, return_ref,
     created_at, updated_at)

Dependency(from_task_id, to_task_id, type: blocks|blocked_by)

ContextSnapshot(id, task_id?, time, app, window_title,
                url?, repo?, branch?, file_path?, note?)

ChangeLog(id, time, actor: user|agent, plan, diff)

CalendarBlock(task_id, start, end)
```

## 9. Архитектура (высокоуровневая)

```
┌─────────────────────────────────────────────────┐
│                  Backend (API + DB)              │
│         FastAPI + SQLite (→ Postgres позже)      │
│                                                  │
│  Tools API ──── Agent Engine ──── Rule Engine    │
│       │              │                │          │
│  CRUD/Search    LLM (next-step,     WIP=3,      │
│  bulk ops       priorities,         distraction  │
│  undo/redo      DoD generation)     rules        │
└──────────┬──────────────┬────────────────────────┘
           │              │
     ┌─────┴─────┐   ┌────┴─────┐
     │  Desktop  │   │ Telegram │
     │  (Tauri/  │   │   Bot    │
     │ Electron) │   │          │
     └─────┬─────┘   └──────────┘
           │
    ┌──────┴──────┐
    │  Extensions │
    │ Cursor+Comet│
    │ + macOS API │
    └─────────────┘
```

- **Единый backend** — desktop и Telegram работают с одними данными.
- **Правила (WIP, хвосты, якоря)** — детерминированные (rule engine), не LLM.
- **LLM** — только для: next-step suggestions, приоритезация, DoD-генерация, массовые агентные операции.
- **Хранилище:** SQLite локально (→ sync с backend позже для Telegram).

## 10. GTD-lite: встроенные практики

| GTD-практика | Адаптация в Wipster |
|---|---|
| Capture (Inbox) | Quick Add (1 строка + Enter) + автоконтекст |
| Clarify (Next Action) | Next step обязателен при Doing/Pause |
| Organize (Projects) | Projects → Queue/Doing/Done + Someday/Maybe |
| Reflect (Weekly Review) | 15–25 мин, с кнопками и агентом |
| Engage (выбор) | Режимы приоритезации (Deadline/Impact/Unblock/Energy) |

## 11. Поведенческий профиль пользователя (основа дизайна)

На основе диагностики из исходного диалога:

- **Старт:** легко на интересном; неприятные задачи — трудно.
- **Завершение:** теряет интерес/переключается → трудно вернуться.
- **Основные срывы:** потеря интереса + переключение на новое.
- **WIP де-факто:** 6–10 одновременно.
- **Режим работы:** нужен deep work, но среда дробит.
- **Ворует фокус:** люди/созвоны + внутренние переключения.
- **Новая задача:** начинает сразу параллельно.
- **Планирование:** хаотично/когда припрёт.
- **Стиль:** свобода, минимум бюрократии.
- **Мотивация:** внутренняя; внешние дедлайны работают как "пинок", но демотивируют.

## 12. MVP (порядок реализации)

1. **Backend + модель данных + tools + changelog/undo.**
2. **Telegram бот** (самый быстрый интерфейс: агент, triage, quick add).
3. **macOS Desktop UI:** Kanban + проекты + WIP=3 + agent panel.
4. **Desktop context observer** + Cursor/Comet extensions.
5. **Distraction detector** + next-step suggestions (только по событию).
6. *(Опционально)* Event-based screenshots → локальный OCR → улучшить подсказки.
